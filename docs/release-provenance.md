# Public Release Provenance

offline-verifiable authorization and execution receipts for AI and high-consequence software actions

## 1. What this provenance covers

This document covers release evidence that is currently produced by this repository and can be verified locally with repository commands.

Current scope:

- release artifact integrity and completeness
- conformance contract and TS/Python parity evidence
- deterministic verifier behavior on canonical fixtures

## 2. Official release artifacts

For a `v*` tag, the official release evidence is:

1. Source tag and tagged source tree (`v*`)
2. Canonical fixture corpus (`fixtures/*.json`, excluding templates)
3. `conformance-manifest.json` (machine-readable expected verifier outcomes)
4. `conformance-snapshot.json` (generated parity evidence from `pnpm conformance`)
5. `verification-summary.json` (generated by `pnpm verify-demo`)
6. `checksums.txt` (SHA-256 checksums from `pnpm checksums`)
7. `release-assets.txt` (artifact inventory generated in release workflow)

These artifacts are collected in `.github/workflows/release.yml` under `release-artifacts/*`.

## 3. What a third party can verify locally

A third party can verify that:

1. The published fixture corpus matches expected checksums.
2. TypeScript and Python verifiers produce semantically matching conformance outputs.
3. Canonical fixtures produce expected pass/fail outcomes under the released verifier.
4. Demo verification accepts a valid chain and rejects a tampered chain.
5. The local release artifact set is complete according to workflow expectations.

## 4. 3-minute verification flow

Prerequisites:

- Node.js 20+
- Python 3.12+
- `pnpm`

Quick flow (after clone):

1. Install dependencies

```bash
pnpm install --frozen-lockfile
python -m pip install -e python/decision_passport_py
```

1. Verify cross-language conformance parity

```bash
pnpm conformance
```

1. Verify valid and tampered demo behavior

```bash
pnpm verify-demo
```

1. Generate local checksum evidence

```bash
pnpm checksums
```

This flow is designed to be executable in about 3 minutes on a typical developer machine once dependencies are available.

## 5. Trust model and limitations

This provenance model verifies artifact integrity and verifier behavior. It does not by itself prove author identity or external notarization.

Current guarantees:

- Offline verification of bundle integrity and conformance behavior.
- Deterministic TS/Python parity checks against canonical fixtures.
- Reproducible local generation of parity and verification evidence.

Current limitations:

- No release signing or cryptographic author identity proof in current scope.
- No external timestamping/notarization guarantees in current scope.
- No claim that bundle contents reflect real-world execution truth beyond recorded artifact integrity.

## 6. Reproducing the evidence locally

Reference path for a specific tag:

```bash
git checkout v0.5.1
pnpm install --frozen-lockfile
python -m pip install -e python/decision_passport_py
pnpm build
pnpm test
pnpm conformance
pnpm verify-demo
pnpm checksums > checksums.local.txt
```

Then compare:

1. `checksums.local.txt` with published `checksums.txt`
2. local `artifacts/conformance-snapshot.json` with published `conformance-snapshot.json`
3. local `artifacts/verification-summary.json` with published `verification-summary.json`

## 7. Relationship to conformance and parity

- `fixtures/conformance-manifest.json` defines expected verifier outcomes.
- `pnpm conformance` validates TS/Python semantic parity and writes `artifacts/conformance-snapshot.json`.
- Verifier outputs are standardized by `docs/verifier-auditor-output.md`.
- Release workflow enforces artifact completeness checks before publishing release assets.

## 8. Non-goals

This provenance surface does not currently provide:

1. release-signing architecture
2. external attestation frameworks
3. supply-chain notarization beyond repository workflow outputs
4. runtime execution attestation outside exported artifacts
